Why AutoCompleteJS?

AutoCompleteJS Genesis

  • "Autocomplete involves the program predicting a word or phrase that the user wants to type in without the user actually typing it in completely.

    Autocomplete ... speeds up human-computer interactions in environments to which it is well suited." -- Wikipedia
  • My team was using Autocomplete on a project and weren't satisfied with our existing solution
  • Other teams were using different Autocomplete libraries and had custom hacks
  • More teams were looking to add Autocomplete components

Existing Autocomplete Components

  • jQueryUI Autocomplete
  • jQuery Tokeninput
  • VisualSearch.js
  • None of these components had exactly what we were looking for.
  • None of these components were easily modified to meet our requirements.

AutoCompleteJS Objectives

  • Feature complete for existing use cases with standard config
  • IE7+ compatibility
  • Faster than alternatives (ie: should be fast in IE7)
  • Minimal dependencies
  • Easy for developers to use

What is AutoCompleteJS?

By the Numbers

  • ~2,500 lines of JS
  • ~100 lines of CSS
  • JS is 20.9 KB minified (for reference, jQuery minified is 90.5 KB)
  • CSS is 2.32 KB minified (for reference, Bootstrap minified is 103.5 KB, jQueryUI Smoothness theme is 26.8 KB)
  • JS code passes the Google Closure Linter 100%
  • Works in IE6+
  • Minimum jQuery version required is 1.4.2 (released Feb 2010)

How does it work?

Design Patterns

  • Uses the constructor pattern to create a new object
  • DOM elements kept in memory
  • Most state kept in memory
  • Most events are handled on the parent element using event delegation
  • Pure functions where possible
  • Most functions that affect state have guard clauses

Code Organization

  • "Module scope" constants (13 variables)
  • "Module scope" stateful (9 variables)
  • Util functions (14 functions)
  • Data validation (9 functions)
  • Data expansion / set defaults (6 functions)
  • Markup building functions (9 functions)
  • DOM manipulation (14 functions)
  • Control flow / data manipulation (30 functions; lots of state management)
  • Browser events (16 functions)
  • Public API methods (18 functions)
  • Initialization (3 functions)

Lessons Learned

Learn From Your Predecessors

  • Someone else has almost certainly tried it before you.
  • "Stand on the shoulders of giants"
  • Learn "Here be dragons" areas
  • Keydown setTimeout example from jQuery Tokeninput

Examples and API

  • Examples and API should be driving the code, not the other way around
  • Examples should be idiomatic: write them the way you want people to use your API
  • API and Examples should link to each other
  • Each example and API endpoint should have a unique URL
  • Your examples become your de facto test cases

Semantic Versioning

  • Fantastically good idea
  • Win-win-win
  • Can be used for internal projects too
  • semver.org

Remove Barriers to Adoption

  • IE7+ compatability, speed
  • Minimal dependencies
  • Lots of examples to start from
  • Plentiful documentation
  • Unique error codes and meaningful error messages
  • Download links for every use case
  • What things have prevented you from using a library / language / platform in the past?

Data is King

  • Nothing is better than pure data
  • "Smart data structures and dumb code works a lot better than the other way around." -- Eric Raymond
  • "It is better to have 100 functions operate on one data structure than to have 10 functions operate on 10 data structures." -- Alan J. Perlis
  • Correctly modeling the data of your program is the hard part; working with that data should be easy.
  • It's a code smell if the complexity of your program is in the way it deals with data and not the data itself.

Misc

  • Pure functions are always correct. Prefer a pure function to any other abstraction.
  • Start with great defaults, then let people override as needed.
  • Prefer explicit comparisons
  • Use a linter
